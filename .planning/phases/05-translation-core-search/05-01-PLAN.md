  <name>Task 1: Create TokenService with CRUD operations using repository pattern</name>
  <files>packages/server/src/service/token.service.ts, packages/server/src/service/project.service.ts</files>
  <action>
Create `packages/server/src/service/token.service.ts` as a NestJS Injectable service. This service extracts all token-related logic from `ProjectService` and rewrites it to use the Drizzle repositories from Phase 2 instead of Mongoose.

**Methods to implement (migrated from ProjectService):**

1. `findByProject(projectId: string)`: Query tokens table WHERE project_id = projectId, ordered by created_at DESC. Include token history by doing a separate query to token_history table joined with users table to get user name/email/avatar. Return tokens with populated history.

2. `findById(tokenId: string)`: Get single token by ID. Include history with user details. Throw NotFoundException if not found.

3. `create(data: { projectId, key, module?, tags?, comment?, translations?, screenshots?, userId, ipAddress?, userAgent? })`:
   - Check for duplicate key within project (SELECT WHERE project_id AND key). Throw BadRequestException if duplicate.
   - Insert into tokens table with JSONB translations column.
   - Insert initial history record into token_history table with userId and translations snapshot.
   - Log activity via ActivityLogService (type: TOKEN_CREATE).
   - Return created token with populated history.

4. `update(tokenId: string, data: { key?, module?, tags?, comment?, translations?, screenshots?, userId, ipAddress?, userAgent? })`:
   - Fetch existing token. Throw NotFoundException if missing.
   - If key is changing, check for duplicate key within same project (exclude current token). Throw BadRequestException if duplicate.
   - For translations: use MERGE semantics -- spread existing translations with new ones (`translations || $new::jsonb` in SQL, or build merged object in service and SET). This is critical: the frontend sends only changed language values, not all languages.
   - If translations changed (diff check), insert new history record into token_history table.
   - Track field changes for activity log (compare old vs new for key, tags, comment, translations).
   - Log activity via ActivityLogService (type: TOKEN_UPDATE) with changes array.
   - Return updated token with populated history.

5. `delete(tokenId: string, userId: string, ipAddress?: string, userAgent?: string)`:
   - Fetch token first to get key name and projectId for logging.
   - Delete from tokens table (cascade will handle token_history via FK ON DELETE CASCADE).
   - Log activity via ActivityLogService (type: TOKEN_DELETE).
   - Return deleted token data.

**Important implementation notes:**
- Inject `TokenRepository` and `TokenHistoryRepository` (from Phase 2), NOT Drizzle directly.
- Inject `ActivityLogService` for audit logging.
- Inject `ProjectService` only for `checkUserProjectPermission()` -- do NOT create circular dependency. If needed, inject the project repository directly instead.
- Use the `diffObject` utility from `src/utils/object` (or rewrite as simple JSON comparison) to detect translation changes before creating history records.
- All IDs are UUIDs (from Phase 2 schema), not MongoDB ObjectIds.

After creating TokenService, **remove** the following methods from `ProjectService`:
- `getProjectTokens()`
- `createToken()`
- `getTokenById()`
- `updateToken()`
- `deleteToken()`
- `batchUpdateTokenModule()` (will be re-added in Plan 03 via TokenService)
- `filterTokensByScope()` (will be re-added in Plan 02 via TokenService)

Keep import/export methods in ProjectService (they are project-level operations). Update them to use TokenService or TokenRepository if they reference `this.tokenModel`.

<task type="auto">
  <name>Task 2: Create TokenController with REST endpoints and register in AppModule</name>
  <files>packages/server/src/controller/token.controller.ts, packages/server/src/controller/project.controller.ts, packages/server/src/app.module.ts</files>
  <action>
Create `packages/server/src/controller/token.controller.ts` as a NestJS Controller. This extracts all token endpoints from `ProjectController` into a dedicated controller.

**Endpoints to implement:**

1. `GET /api/tokens/:projectId` -- List all tokens for a project
   - `@UseGuards(AuthGuard)`, `@CurrentUser() user: UserPayload`
   - Verify permission via `projectService.checkUserProjectPermission(projectId, user.userId)` -- throw ForbiddenException if unauthorized
   - Call `tokenService.findByProject(projectId)`
   - Return token array

2. `POST /api/tokens` -- Create a token
   - Body: `{ projectId, key, module?, tags?, comment?, translations?, screenshots? }`
   - Verify permission on projectId
   - Call `tokenService.create({ ...data, userId: user.userId })`
   - Return created token

3. `GET /api/tokens/detail/:tokenId` -- Get single token with history
   - Verify permission by looking up token's projectId first
   - Call `tokenService.findById(tokenId)`
   - Return token with history

4. `PUT /api/tokens/:tokenId` -- Update a token
   - Body: `{ key?, module?, tags?, comment?, translations?, screenshots? }`
   - Fetch token to get projectId, verify permission
   - Call `tokenService.update(tokenId, { ...data, userId: user.userId })`
   - Return updated token

5. `DELETE /api/tokens/:tokenId` -- Delete a token
   - Fetch token to get projectId, verify permission
   - Call `tokenService.delete(tokenId, user.userId)`
   - Return success

6. `GET /api/tokens/:tokenId/history` -- Get token change history
   - Fetch token to get projectId, verify permission
   - Call `tokenService.findById(tokenId)` and return just the history array
   - Return array of history entries with user details

**After creating TokenController, remove the following endpoints from ProjectController:**
- `GET /api/project/tokens/:projectId`
- `POST /api/project/token`
- `GET /api/project/token/:tokenId`
- `PUT /api/project/token/:tokenId`
- `DELETE /api/project/token/:tokenId`
- `POST /api/project/tokens/batch/module` (will be re-added in Plan 03)

Keep export/import/preview endpoints in ProjectController (they are project-level).

**Register in AppModule:**
- Add `TokenController` to the controllers array in `app.module.ts`
- Add `TokenService` to the providers array in `app.module.ts`
- Ensure `TokenRepository` and `TokenHistoryRepository` are available (should already be from Phase 2's database module)

**Permission pattern:** Every endpoint follows the same pattern:
1. Extract user from JWT via `@CurrentUser()`
2. Determine projectId (from param, body, or by looking up the token)
3. Call `checkUserProjectPermission(projectId, user.userId)`
4. Throw ForbiddenException if false
