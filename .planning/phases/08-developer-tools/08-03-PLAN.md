    Create a comprehensive REST API reference document at `docs/api-reference.md`. The document must be derived from the actual controller source code -- read each controller file to extract exact endpoint paths, methods, parameters, and guards.

    **Structure the document as follows:**

    1. **Header and Introduction:**
       - Title: "qlj-i18n REST API Reference"
       - Base URL: `http://localhost:3001` (configurable via PORT env var)
       - API version note: "This documents the current API surface. No versioned prefix is used yet."

    2. **Authentication section:**
       - **JWT Authentication:** POST /api/auth/login -> get token -> use as `Authorization: Bearer <jwt>`
       - **API Key Authentication:** Generate from web UI or POST /api/api-keys -> use as `Authorization: Bearer qlji_<key>`
       - Include curl examples for both methods
       - Note: Most endpoints accept both JWT and API key

    3. **Endpoint sections** (one section per controller, with a table of contents at top):

       For each endpoint, document:
       - HTTP Method + Path
       - Authentication: Required (JWT/API key) or Public
       - Request body (with TypeScript-style type annotation)
       - Response format (success case with example JSON)
       - Error responses (common ones: 401, 403, 404)

       **Sections to include:**
       a. **Authentication** (`/api/auth/*`) -- register, login, status, token generation
       b. **Users** (`/api/user/*`) -- current user, search, get by ID
       c. **Teams** (`/api/team/*`) -- read the team controller to document all endpoints
       d. **Projects** (`/api/project/*`) -- CRUD, languages, modules
       e. **Translation Tokens** (`/api/project/token*`) -- CRUD, batch operations
       f. **Import & Export** (`/api/project/import/*, /api/project/export/*, /api/project/download/*`) -- file operations
       g. **API Keys** (`/api/api-keys/*`) -- create, list, revoke
       h. **AI Translation** (`/api/ai/*`) -- read the AI controller to document endpoints
       i. **Activity Logs** (`/api/activity-logs/*`) -- read the controller to document endpoints

    4. **MCP Server section:**
       - Connection URL: `http://localhost:3001/api/mcp`
       - Authentication: API key required in Authorization header
       - Claude Desktop / Cursor config example:
         ```json
         {
           "mcpServers": {
             "qlj-i18n": {
               "url": "http://localhost:3001/api/mcp",
               "headers": {
                 "Authorization": "Bearer qlji_your_api_key_here"
               }
             }
           }
         }
         ```
       - Available tools table: name, description, parameters
       - The 5 tools: list_projects, list_project_tokens, get_token_details, create_token, update_token

    5. **CLI Quick Reference section:**
       - Installation: `npm install -g qlj-i18n`
       - Commands table: login, init, pull, push with brief description and example
       - Environment variables: `QLJ_I18N_API_KEY`, `QLJ_I18N_SERVER`

    **IMPORTANT:** Read the actual controller source files to get exact routes, parameter names, and types. Do NOT guess or use outdated information. The controllers to read:
    - `packages/server/src/controller/auth.controller.ts`
    - `packages/server/src/controller/user.controller.ts`
    - `packages/server/src/controller/team.controller.ts`

    - `packages/server/src/controller/mcp.controller.ts`

    **Style:**
    - Use Markdown with code blocks for JSON examples
    - Use tables for parameter lists
    - Keep descriptions concise but complete
    - Include at least one curl example per major endpoint group
