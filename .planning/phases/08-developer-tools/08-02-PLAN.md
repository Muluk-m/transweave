  <action>
    1. **Modify McpController** (`packages/server/src/controller/mcp.controller.ts`) to require API key auth:
       - Inject `ApiKeyService` via constructor
       - In the `handleMcp` method (the `@All()` handler):
         a. Before creating/reusing transport, extract the Authorization header
         b. If method is GET for the `/api/mcp/info` endpoint, skip auth (keep info page public)
         c. For all other requests, validate API key:
            ```
            const authHeader = req.headers.authorization;
            if (!authHeader?.startsWith('Bearer qlji_')) {
              return res.status(401).json({ jsonrpc: '2.0', error: { code: -32000, message: 'API key required. Use Authorization: Bearer qlji_...' } });
            }
            const apiKeyResult = await this.apiKeyService.validateKey(authHeader.split(' ')[1]);
            if (!apiKeyResult) {
              return res.status(401).json({ jsonrpc: '2.0', error: { code: -32001, message: 'Invalid or expired API key' } });
            }
            ```
         d. Store user context on the request object: `req['mcpUser'] = apiKeyResult;`
         e. Pass user context to McpService so tools can use the real userId

    2. **Modify McpService** (`packages/server/src/service/mcp.service.ts`) to support user context and add update_token:
       - Add a method to set user context that the controller calls before tool execution:
         `setUserContext(userId: string)` -- stores on instance or uses a per-request approach
         Since MCP uses sessions, store userId in a Map keyed by sessionId
       - **Refactor tool registration** to accept user context:
         - Change the `registerTools()` approach so that each tool handler can access the current user's ID
         - The simplest approach: McpService stores a `sessionUserMap: Map<string, string>` mapping sessionId to userId
         - Add methods: `setSessionUser(sessionId: string, userId: string)` and `getSessionUser(sessionId: string): string`
         - Controller calls `setSessionUser` after auth validation, passing the transport's sessionId
       - **Update create_token tool**: Replace hardcoded `'000000000000000000000000'` userId with `this.getSessionUser(sessionId)` or equivalent
       - **Add update_token tool**:
         - Schema (Zod):
           ```
           z.object({
             tokenId: z.string().describe('Token ID to update'),
             key: z.string().optional().describe('Updated token key'),
             translations: z.record(z.string()).optional().describe('Updated translations { "lang": "text" }'),
             module: z.string().optional().describe('Updated module code'),
             tags: z.array(z.string()).optional().describe('Updated tags'),
             comment: z.string().optional().describe('Updated comment'),
           })
           ```
         - Handler calls `this.projectService.updateToken(params.tokenId, { ...params, userId })` using authenticated user's ID
         - Returns updated token as JSON text content
       - **Update tool descriptions to English** (current descriptions are in Chinese):
         - list_projects: title "List Projects", description "List all i18n projects"
         - list_project_tokens: title "List Project Tokens", description "List tokens for a specific project"
         - get_token_details: title "Get Token Details", description "Get detailed information about a token"
         - create_token: title "Create Token", description "Create a new translation token"
         - update_token: title "Update Token", description "Update an existing translation token"
       - **Update MCP info page** to reflect auth requirement: Add a note that API key is required and update the connection config example to include the header

    7. **login.ts** (`packages/cli/src/commands/login.ts`):
       - Command: `qlj-i18n login`
       - Options: `--server <url>` (server URL), `--api-key <key>` (API key)
       - Both options are required (no interactive prompts for v1)
       - Validates API key by calling `GET /api/auth/status` with the key
       - On success: saves server and apiKey to global config, prints "Logged in successfully. Config saved to ~/.config/qlj-i18n/config.json"
       - On failure: prints error and exits with code 1

    8. **init.ts** (`packages/cli/src/commands/init.ts`):
       - Command: `qlj-i18n init`
       - Options: `--project-id <id>` (required), `--output-dir <dir>` (default: `./src/locales`), `--format <fmt>` (default: `json`)
       - Validates project exists by calling `GET /api/project/find/:projectId`
       - On success: creates `.qlj-i18n.json` with `{ projectId, outputDir, format, languages }` (languages from project response)
       - Prints "Initialized qlj-i18n config for project: <name>"

    9. **pull.ts** (`packages/cli/src/commands/pull.ts`):
       - Command: `qlj-i18n pull`
       - Options: `--format <fmt>` (override config), `--output <dir>` (override config), `--languages <langs...>` (override config, comma-separated)
       - Load project config from `.qlj-i18n.json`; fail if not found ("Run qlj-i18n init first")
       - Load global config for server/apiKey; fail if not found ("Run qlj-i18n login first")
       - Fetch project info: `GET /api/project/find/:projectId` to get available languages
       - For each language (from options or config or project):
         - Call `GET /api/project/download/:projectId?format=<fmt>&languages=<lang>` (this returns a ZIP)
         - Extract the single language file from the ZIP
         - Write to `<outputDir>/<lang>.<format>` (e.g., `./src/locales/en.json`)
         - Print `  Downloaded <lang> -> <outputPath>`
       - IMPORTANT: The download endpoint returns a ZIP file. Use Node's built-in zlib or a simple ZIP extraction. Since the project already uses `jszip` in the server, consider using the same approach (but CLI should stay light). Alternative: the server already has individual export capability in the ZIP -- unzip and extract the file.
       - Simplest approach for v1: Use the download endpoint which returns a ZIP, then use `node:zlib` with basic ZIP parsing (or add jszip as a dependency to the CLI). jszip is already a dependency in the monorepo.
       - Actually, the simplest approach: add jszip to CLI dependencies and use it to extract files from the ZIP response buffer.

    10. **push.ts** (`packages/cli/src/commands/push.ts`):
        - Command: `qlj-i18n push`
        - Options: `--format <fmt>` (override config), `--input <dir>` (override config outputDir), `--languages <langs...>` (override config)
        - Load project config and global config (same validation as pull)
        - Scan `<inputDir>` for files matching `<lang>.<format>` pattern
        - For each language file found:

          - Call `POST /api/project/import/:projectId` with `{ language: <lang>, content: <fileContent>, format: <fmt>, mode: 'append' }`
          - Print result: `  Pushed <lang>: <added> added, <updated> updated, <unchanged> unchanged`
        - Print summary: "Push complete: <N> languages processed"
